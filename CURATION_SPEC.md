# AI 작업실 — 현상설계 실무 AI 도구·기법 큐레이션 명세

이 파일은 자동 큐레이션 작업의 단일 기준 문서. 스케줄 작업이 매 회차 이 명세를 따른다.
변경 시 이 파일만 수정하면 다음 회차부터 반영됨.

당선 사이트 `/studio` (메뉴 "AI 작업실") 로 게재. `/curation`(연구·코드 모델)·`/trends`(공고)와
**겹치지 않는다** — 여긴 *손에 잡히는 실무 도구*만.

---

## 0. 사용자 도메인

- 건축가 / **현상설계·설계공모 실무자**. 빌더가 아니라 *만드는 사람* (디자인·캐드·일러·라이노·GH·렌더·보고서).
- 관심: **이번에 나온/업데이트된 AI 도구·기법 중, 현상설계 결과물을 실제로 더 빨리·낫게 뽑게 해주는 것.**
- 본인 파이프라인: AutoCAD · Illustrator(+ `cad_to_ai_jsx` 자동화) · Rhino · Grasshopper · 렌더/CG · 보고서.
- 데모·연구물·"AI가 건축을 바꾼다" 류 일반론 ❌ / 다음 주 패널 한 장·렌더 한 컷·다이어그램 한 개를 실제로 돕는 것 ✅.

## 1. 절대 필터 — "현상설계 결과물 하나를 실제로 개선하는가?"

각 아이템은 다음 질문을 통과해야 함:

> **이 도구/기법이 현상설계 산출물 — 컨셉·다이어그램 / 매싱 / 라이노·GH 모델 / 도면 / 렌더·CG / 패널 레이아웃 / 보고서 — 중 하나를 실제로 더 빨리 또는 낫게 만드는가?**

통과 ✅:
- 상용·무료 AI 도구 (Veras, D5 Render AI, Krea, Magnific, PromeAI, LookX, Maket, ArkoAI 등) — *데모여도 실제 출시·체험 가능*하면 OK.
- 라이노/그래스호퍼 플러그인·스크립트 (Food4Rhino 신규/업뎃, GH 정의, ML 폼파인딩).
- 워크플로우·튜토리얼 (ControlNet 건축 셋업, 스케치→렌더, plan→3D, 일러 generative fill 활용법).
- 보고서·패널·도면 자동화 (설계설명서 초안, 레이아웃 자동, 도면 캡션).
- 실사용 후기·비교 (커뮤니티에서 "이거 실무에 써보니" 류).

자동 제외 ❌:
- "AI가 건축을 바꾼다" 식 일반 전망·칼럼·인터뷰.
- 순수 연구 논문·ML 모델 가중치만 (← 이건 `/curation` 영역. 단, *건축CG에 바로 써먹을 수 있게 패키징된* 이미지→3D·업스케일 등은 OK).
- 벤더 투자·인수·실적 뉴스, 컨퍼런스 홍보.
- 시공·부동산·스마트빌딩·BIM 운영 등 *설계공모 산출물과 무관한* 건축 AI.
- 코딩 일반 도구 (← `/news` 영역).

**도감 프레임 (핵심):** 이 피드는 "오늘의 뉴스"가 아니라 **실무 도구 도감**이다. 우물은 `scripts/tools-catalog.json` 의 실제 도구 카탈로그 — 매일 거기서 *아직 최근 안 다룬* 도구를 골라 실무 각도로 소개(스포트라이트)하고, 수집 소스에 *새로 뜬* 도구·업데이트가 있으면 위에 얹는다. 그래서 **마른 날이 없다** — 카탈로그가 바닥을 받쳐 매일 충실하다. 단 지어내기·과장 금지: 모르는 가격·기능은 단정 말고 "확인 필요". 직전 회차와 같은 도구 중복 금지(카탈로그 로테이션).

## 2. 소스 — 카탈로그(우물) + 수집처(신상)

- **`scripts/tools-catalog.json`** — 실제 도구 카탈로그. 도감의 **우물**. 매일 여기서 스포트라이트를 골라 매 호의 바닥을 채운다. 새 도구 발견 시 여기 추가(이름·url·stage·blurb).
- **`scripts/sources.json`** — 그날 *새로 뜬 것*을 잡는 수집처. 카탈로그 위에 신상을 얹는 역할.

수집처는 *실무자가 실제로 도구를 발견하는 곳* + 주요 제품 블로그. 공고 포털·GitHub trending 아님.

- 제품 블로그 RSS: **D5 Render · EvolveLAB(Veras) · ShapeDiver** — 도구 자체의 신버전·AI 기능 1차 소스.

- **McNeel Discourse** (discourse.mcneel.com/latest.json) — 라이노·GH 실사용·신규 플러그인 1순위.
- **Reddit**: r/architecture · r/rhino · r/grasshopper3d · r/StableDiffusion(건축) — 실무자 도구 발견·후기.
- **Parametric Monkey** (RSS) — 파라메트릭/GH 튜토리얼·도구.
- **ArchDaily** (RSS) — 건축 일반에서 AI 도구·기법 기사만 골라냄 (필터가 거름).
- **Hacker News Algolia** — "architecture render" · "rhino grasshopper" 등 도구 신호.
- **arXiv cs.GR** (RSS, 가볍게) — 건축CG에 바로 써먹을 수 있게 데모/코드까지 나온 것만.

소스 추가·교체는 `scripts/sources.json` 만 수정. `kind`: `json`(Reddit/Discourse/HN) · `rss`(Parametric Monkey/ArchDaily/arXiv) · `html`(기본).

## 3. 등급 (score 1~10) — "실무 투입 가능성"

요한님 현상설계 실무 관점의 *지금 당장 써먹힘*. 종합 판단:
- 다음 공모 산출물(렌더/다이어그램/모델/패널/보고서)에 바로 투입 가능 (가산).
- 무료 또는 합리적 가격, 한글/한국 환경에서 무리 없음 (가산).
- 본인 파이프라인(라이노·일러·`cad_to_ai_jsx`)에 바로 붙음 (가산).
- 데모만 있고 실제 못 씀 / 가격·접근 장벽 큼 / 결과 품질 낮음 (감산).

- ★ 9~10 — 지금 당장 다음 작업에 투입할 직격 도구
- ★ 6~8 — 시도 가치 있음, 한 번 테스트해볼 만
- ★ 3~5 — 흐름상 알 가치, 직접 투입은 아직
- ★ 1~2 — 참고용 (초기 데모·연구)

## 4. 출력 분량·섹션 (파이프라인 단계별)

- **pick (오늘 눈에 띈 것)**: 1~3개 — 그날 최고 1순위. 다른 섹션과 중복 금지.
- **concept (컨셉·다이어그램)**: 0~4개 — 아이디어/매싱/다이어그램 생성.
- **modeling (모델링 — 라이노/GH/CAD)**: 0~4개 — 파라메트릭/모델링/도면 자동화.
- **visual (렌더·CG·이미지)**: 0~5개 — 렌더/업스케일/스케치→이미지/스타일.
- **panel (패널·보고서)**: 0~3개 — 레이아웃/일러·인디/문서 자동화.
- **technique (기법·워크플로우)**: 0~2개 — 이번에 따라 해볼 워크플로우 1개.
- 전체 **7~10개로 충실하게** (도감이라 카탈로그로 매일 채움 — 마른 날 없음). 단계 골고루. 지어내기·과장은 금지.

## 5. 언어 규칙

- 해요체/합니다체 섞어 간결하게. 과장·영업체 금지. **데모면 데모라고, 별로면 별로라고** 쓴다.
- 건축 약어(RFP·SD·DD·렌더·매싱·다이어그램·플러그인 등) 그대로 OK. AI 약어(ControlNet·LoRA·diffusion 등) 처음 등장 시 짧은 비유 한 줄.
- 가격(무료/유료/구독)·플랫폼(Windows/웹/Rhino8 등)·성숙도(정식/베타/연구 데모)는 **빠짐없이**. 수치는 원문 그대로.
- 슬랭("박다"·"꽂다") 금지 — 공개 콘텐츠.

### 항목 구성 (.ni 카드 — news·curation·trends 와 동일 폼)

- **body** (3~4문장): 무슨 도구·기법인지, 현상설계 어느 단계에 닿는지, 비슷한 도구에 빗대 한 줄.
- **points** (불릿 2~3): 가격·플랫폼 / 입력→출력 (예: 스케치 → 렌더) / (선택) 한계·주의.
- **한줄평** (gain): 정직한 한 줄 — *써먹히나, 데모인가, 누구에게.*
- **써먹기** (todo): 현상설계 *어느 산출물에 어떻게* 투입하는지 구체적으로 (패널/렌더/모델/보고서). 가능하면 링크·시작 방법. 요한님 파이프라인(라이노·일러·`cad_to_ai_jsx`)에 붙는 지점이 있으면 명시.
- **성숙도** (why_now): 정식 출시/베타/연구 데모 + 가격 신호 + 왜 지금 볼 만한지.

### 톤 점검 셀프 질문 (작성 후)

1. 실무자가 1초에 *"이걸 내 다음 작업에 쓸까 말까"*를 판단할 수 있나?
2. 가격·플랫폼·성숙도가 빠짐없이 들어갔나?
3. '써먹기'가 추상("워크플로우 개선")이 아니라 *구체적 산출물*("패널 배경 렌더를 5분에")을 짚나?
4. 데모/연구물을 정식 도구처럼 과장하지 않았나?

→ 모두 ✓ 아니면 다시 쓴다.

## 6. 파일 출력

- 경로: `arch-ai-tools-curation/YYYY-MM-DD_요일.md` (.ni 시맨틱 HTML)
- frontmatter: title / eyebrow / hero_title / description / summary
- 섹션 순서: 이번 회차 흐름(intro) → pick → concept → modeling → visual → panel → technique → 맺음말(outro)
- `index.md` 는 워커가 매 회차 자동 갱신

## 7. 일정

- **매일 09:30 KST** (claude -p 로컬 생성, 정액제). `/news`(08:00 계열)·`/trends`(월 09:00)와 시간 분리.
- 빈도 변경 시: 이 파일과 스케줄 작업(run-daily 미러) 둘 다 수정.

## 8. 중복 방지

- 직전 2회차의 최신 파일 2개를 읽고, 동일 URL은 새 파일에서 제외.
- 단, 의미 있는 *신규 버전/업데이트*면 "[업데이트]" 라벨로 재등재 OK.

## 9. 원문 대조 검증

- 생성 후 각 항목 URL을 실제 fetch → 가격·기능·플랫폼·성숙도를 원문과 대조해 정정.
- 원문 못 가져온 항목은 `verified=false` (✓ 원문 대조 배지 미표시).
